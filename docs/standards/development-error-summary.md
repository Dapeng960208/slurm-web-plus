# 开发错误复盘标准（AI 必须遵守）

本标准用于约束 AI（以及开发者）在开发过程中遇到错误时的记录方式，目的是把“踩坑经验”沉淀为可检索的文档资产，避免同一个错误被反复踩到。

本标准适用于内部维护型文档体系，不影响 Antora/手册体系（`docs/modules/`、`docs/man/` 等）。

## 1. 触发条件（什么时候必须记录）

开发过程中出现以下任一情况，必须做错误复盘并写入错误库（见第 2 节）：

- 执行命令失败（构建、测试、迁移、脚本、CI 相关）
- 依赖/环境不一致导致的失败（Python/Node 版本、虚拟环境、权限、路径等）
- 配置缺失或配置错误导致的失败（`*.ini`、策略文件、环境变量等）
- API/路由/权限不一致导致的失败（404/401/403/501 等）
- 文档与实现不一致导致的误用或返工

不需要记录的情况：

- 明显的拼写错误且可在 1 分钟内解决、且没有复现价值的误输入

## 2. 固定入口（写到哪里）

错误复盘必须写入：

- `docs/tracking/error-log.md`

约束：

- 不要把错误只留在聊天记录或提交信息里。
- 错误库以“可检索、可复用”为目标，尽量写清可复现条件与根因。

## 3. 记录模板（必须包含的字段）

每条错误记录至少包含：

- 时间：`YYYY-MM-DD`
- 现象：最关键的报错信息或症状（不要整段粘贴日志，摘关键行）
- 解决办法：最终采用的修复方式（涉及文件、配置或命令时写出关键点）

推荐格式：

```md
### 2026-05-08：一句话概括
- 时间：2026-05-08
- 现象：Dashboard 视图测试中，`wrapper.text()` 断言 `RouterLink` 文本误报失败。
- 解决办法：改为断言稳定 DOM 标记或按钮节点，不再依赖 `wrapper.text()`。
```

## 4. 与其他文档的关系

- 若错误暴露出“规范缺口”，需要同步更新：
  - `docs/standards/documentation-standard.md`
  - 或本标准自身
- 若错误属于“功能行为/接口/配置/权限/迁移”的误用，应同步更新对应正式文档（`docs/overview/`、`docs/guides/`、`docs/features/`）。
